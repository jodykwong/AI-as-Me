## 一、项目愿景

> **个人 AI 代理系统，具有灵魂、记忆和自我进化能力。**

基于"AI养蛊"方法论（余一），项目核心目标应实现：

1. **AI-as-Me**: AI能代表"我"做事，具备个性化人格
2. **自我进化**: AI能从经验中学习，自主生成新规则
3. **能力补充**: Soul不足时，借助外部方法论（如BMad）补充能力
4. **复利工程**: 每次迭代比上一次更好

| 工具 | 定位 | 优势 |

| 工具 | 配置目录 | 当前状态 | 说明 |

| **OpenCode** | `.opencode/` | ❌ 缺失 | MVP工具但未集成 |

**问题**：MVP策略声称支持OpenCode + Claude Code，但实际只有Claude Code的适配配置。

### 3.3 OpenCode集成建议

```
.opencode/
├── config.yaml              # OpenCode全局配置
│   # model: anthropic/claude-4-sonnet
│   # temperature: 0

✅ **合理性**：两个工具覆盖"Open Code"范式的主流方向，互补性好

✅ **扩展性**：其他适配目录已预留，后期扩展成本低

| 优先级 | 差距描述 | 严重程度 | 状态 |
|--------|----------|----------|------|
| **P0** | 缺少"自创规则"回写机制 | 🔴 核心缺失 | 未实现 |
| **P0** | 经验→规则的进化闭环未实现 | 🔴 核心缺失 | 未实现 |
| **P1** | OpenCode集成配置缺失 | 🟡 MVP不完整 | 未实现 |
| **P1** | _bmad未迁移为Skills架构 | 🟡 架构不清 | 待迁移 |
| **P1** | Skills调用机制不明确 | 🟡 逻辑不清 | 待明确 |

| **P2** | 进化日志缺失 | 🟢 可后补 | 未实现 |

### 4.2 P0差距详解

#### 差距1：Soul是只读的

```
当前状态：
┌──────────┐         ┌──────────┐
│  人类    │ ──写入──►│  soul/   │──读取──► AI
└──────────┘         └──────────┘
                     （静态，不变）

目标状态：
┌──────────┐         ┌──────────────────────┐
│  人类    │ ──写入──►│  soul/               │
└──────────┘         │  ├── profile/        │
                     │  ├── rules/core/     │──读取──► AI
┌──────────┐         │  └── rules/learned/  │          │
│   AI     │ ──写入──►│      (自创规则)     │◄─────────┘
└──────────┘         └──────────────────────┘
                     （动态进化）
```

**问题本质**：AI无法将学到的经验固化为自己的规则，每次都是"从零开始"。

#### 差距2：经验只用于检索，不用于进化

```
当前状态：
experience ──► RAG检索 ──► 辅助回答
              （消费经验，单向流动）

目标状态：
experience ──► 模式识别 ──► 规则生成 ──► soul/rules/learned/
              （生产能力，闭环进化）
```

**问题本质**：Agentic RAG只是"查过去"，不是"从过去成长"。

### 4.3 P1差距详解

#### 差距3：_bmad调用机制不清

当前问题：
- _bmad/目录存在，但何时调用不明确
- 没有"soul能力不足"的判断逻辑
- 调用后的结果如何反馈不清晰

应有的逻辑：
```
用户请求 ──► 检查soul能力 ──┬──► 足够 ──► 直接执行
                           │
                           └──► 不足 ──► 调用_bmad补充
                                              │
                                              ▼
                                        执行并记录经验
```

#### 差距4：缺少灵感池机制

余一方法论关键点：
> "把重要但不紧急的想法丢到灵感库，让AI在闲时自主探索"

当前kanban只有任务流（inbox→done），缺少：
- 探索性任务区
- AI自主发起的改进提案
- 闲时学习/探索机制

## 六、Skills功能整合建议

### 6.1 为什么需要Skills

Claude Skills是Anthropic推出的能力封装机制，其核心价值：
- **能力装箱**：把任务SOP、脚本、参考资料放进标准化文件夹
- **按需加载**：只在需要时加载完整内容，节省token
- **可版本化**：能力模块可测试、可复用、可分发

Skills与AI-as-Me的关系：
- **Skills = 外部能力**（静态、人类定义）→ 对应当前的 `_bmad/`
- **Soul = AI自我**（动态、可进化）→ 对应当前的 `soul/`

### 6.2 当前_bmad与Skills的映射

```
当前结构                          Skills架构
─────────────────────────────────────────────────────
_bmad/                    →      skills/bmad/
├── (方法论内容)                 ├── SKILL.md        # 技能定义
└── (工作流模板)                 ├── scripts/        # 可执行脚本
                                 └── references/     # 参考资料

_bmad-output/             →      （执行产物，保持不变）
```

### 6.3 建议的Skills目录结构

```
AI-as-Me/
├── soul/                        # AI的"自我"（不变）
│   ├── profile/
│   ├── mission/
│   └── rules/
│       ├── core/                # 人类定义的核心规则
│       └── learned/             # 🆕 AI自创的规则
│
├── skills/                      # 🆕 外部能力模块
│   ├── bmad/                    # BMad Method（从_bmad迁移）
│   │   ├── SKILL.md             # 技能元数据和指令
│   │   ├── scripts/             # 工作流脚本
│   │   └── references/          # 方法论文档
│   │
│   ├── task-management/         # 任务管理技能
│   │   ├── SKILL.md
│   │   └── scripts/
│   │
│   └── evolution/               # 🆕 自进化技能（养蛊核心）
│       ├── SKILL.md             # 进化逻辑定义
│       ├── scripts/
│       │   ├── pattern_analyzer.py
│       │   └── rule_generator.py
│       └── references/
│           └── evolution_rules.md
│
├── experience/                  # 🆕 经验库（进化的原材料）
│   ├── successes/               # 成功案例
│   ├── failures/                # 失败教训
│   └── patterns/                # 识别出的模式
│
└── kanban/                      # 任务管理（不变）
```

### 6.4 Skills与Soul的关系

```
┌─────────────────────────────────────────────────────────────────┐
│                    Skills vs Soul 架构                          │
├─────────────────────────────────────────────────────────────────┤
