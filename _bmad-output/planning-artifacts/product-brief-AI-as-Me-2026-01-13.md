# Product Brief: AI-as-Me v2.0

## Executive Summary

AI-as-Me v2.0 是一个开源的自进化 AI 数字分身系统，专为技术型创业者打造真正能"替你打工"的智能伙伴。与传统 AI 工具的"用完即弃"模式不同，AI-as-Me 通过独创的"养蛊"机制实现持续进化，让 AI 分身越用越懂你，越用越强。

**核心价值主张：** 从"每次都要重新解释"到"AI 真正懂我"，通过自进化循环让数字分身成为你的长期伙伴而非一次性工具。

---

## Core Vision

### Problem Statement

技术型创业者面临一个核心矛盾：他们需要身兼多职（商业分析、技术开发、市场运营），但时间精力有限。现有 AI 工具虽然智能，却无法真正"替人打工"：

- **无持久记忆**：每次对话都要重新解释背景和偏好
- **无学习能力**：AI 永远停留在"新手"状态，无法积累经验
- **无主动性**：只能被动回应，无法主动推进长期目标
- **无个性化**：通用模型，不了解用户的决策风格和工作习惯

### Problem Impact

- **效率损失**：每次与 AI 交互需要 20+ 分钟解释上下文
- **能力瓶颈**：AI 无法承担复杂的长期任务
- **机会成本**：创业者被琐事消耗，无法专注高价值工作
- **规模受限**：个人能力无法复制，业务增长受制于时间

### Why Existing Solutions Fall Short

| 现有方案 | 核心缺陷 |
|---------|---------|
| ChatGPT/Claude | 会话级记忆，无法积累长期经验 |
| AutoGPT/AgentGPT | 不稳定，无个性化，缺乏进化机制 |
| 个人知识库 + RAG | 只能检索，不能行动和学习 |
| Open Interpreter | 无长期记忆，每次都是"新手" |

**根本缺失：** 缺乏"灵魂"（用户的履历、价值观、决策风格）和"进化"（从经验中学习的能力）

### Proposed Solution

AI-as-Me v2.0 通过三大核心机制解决这些问题：

**1. 灵魂注入系统 (Soul Injection)**
- `profile.md`：用户履历、技能栈、性格特征
- `rules.md`：从经验中提炼的决策法则（自动增长）
- `mission.md`：长期使命和目标

**2. 养蛊进化循环 (Self-Evolution Loop)**
```
任务执行 → 记录日志 → 反思总结 → 规则提取 → 更新规则库 → 下次更聪明
```

**3. 不造轮子，只做编排**
- 调度现有成熟的 Agent CLI 工具（OpenCode、Claude Code、Gemini CLI）
- 通过 Soul 注入和 BMAD 技能扩展实现个性化
- Agentic RAG 实现经验复用和上下文增强

### Key Differentiators

1. **自进化机制** - 独创的"养蛊"循环，让 AI 从每次任务中学习，越用越强
2. **灵魂系统** - 基于用户履历和经验的个性化决策，不是通用模型
3. **编排优先** - 不重复造轮子，调度最优秀的现有 Agent 工具
4. **具身优先** - 优先部署在 RDK X5，实现数字智能 + 物理执行
5. **开源核心** - MIT/Apache 2.0 协议，社区友好，商业化通过价值交付

## Target Users

### Primary Users

**用户画像：技术型单人 AI 创业者**

**代表人物：Jody**
- **背景**：Python 开发者，专注 AI 领域，拥有 PMP 认证
- **创业状态**：单人创业，身兼多职（开发者、产品经理、市场、商业分析）
- **技术栈**：Python、AI/ML 开发、机器人/嵌入式（RDK X5）、FastAPI、ROS2
- **硬件环境**：拥有 RDK X5 机器人（XLeRobot），希望实现具身 AI

**日常工作挑战：**
- 📊 需要做 AI 市场调研发现创业机会，但写代码已占据大部分时间
- 💻 开发 AI 产品和原型，同时要处理运营和商业分析
- 📈 产品推广和用户增长，缺乏专门的市场团队
- 🔍 评估市场和竞争态势，但每次使用 AI 工具都要重新解释背景

**核心痛点：**
- **时间分散**：需要扮演多个角色，难以专注于高价值工作
- **重复解释**：每次使用 ChatGPT 等工具都要花 20+ 分钟解释项目背景、偏好、输出格式
- **无法积累**：AI 工具无法记住用户的决策风格和工作习惯
- **缺乏主动性**：现有工具只能被动回应，无法主动推进长期目标

**成功愿景：**
- 拥有一个真正"懂我"的 AI 分身，无需反复解释背景
- AI 分身能产出经得起推敲的高质量商业分析报告
- 规则库不断积累，分身越用越强
- 将 80% 时间专注于核心技术开发，其他工作交给分身

### Secondary Users

**潜在次要用户群体：**

| 用户类型 | 特征 | 需求差异 |
|---------|------|---------|
| **小型 AI 团队技术负责人** | 2-5 人团队，需要提升效率 | 可能需要多用户协作功能 |
| **AI 爱好者/学习者** | 技术背景，想尝试 AI Agent | 更注重学习和探索过程 |
| **开源社区贡献者** | 开发者，对"养蛊"机制感兴趣 | 贡献代码、扩展功能 |
| **机器人研究者** | 拥有 RDK X5 等硬件平台 | 关注具身 AI 和硬件集成 |

### User Journey

**典型用户旅程：从"身兼多职"到"有人替我打工"**

**1. 发现阶段 (Discovery)**
- 通过技术博客/播客听说"AI 养蛊"或"AI 数字分身"概念
- 被"越用越强"的进化机制吸引
- 在 GitHub 发现 AI-as-Me 开源项目

**2. 首次使用 (Onboarding)**
- 按照 SOP 在 RDK X5 上部署系统
- 创建"灵魂文件"（profile.md、rules.md、mission.md）
- 在看板中创建第一个任务：市场调研或技术分析

**3. 核心使用 (Core Usage)**
```
每日循环：
1. 早上：在 inbox 中投放新任务
2. AI 分身：读取任务 → 结合 profile 分析 → 执行 → 生成报告
3. 晚上：查看 done 目录的完成结果
4. 定期：AI 自动反思更新 rules.md
```

**4. "Aha！"时刻 (Value Realization)**
- 第一次看到分身完成的分析报告，结果经得起推敲
- 发现 rules.md 自动积累了有价值的经验法则
- 意识到分身真的"懂我"了，不再需要反复解释

**5. 长期价值 (Long-term)**
- 分身成为日常工作的一部分
- 规则库成为个人知识资产
- 从"省时间"升级为"扩展能力边界"
- 考虑将成功经验分享给社区

## Success Metrics

### User Success Metrics

**核心成功标准：结果采纳率 >80%**

AI 分身产出的结果（商业分析报告、市场调研等）被用户直接采纳或仅需少量修改即可使用的比例超过 80%，即视为"经得起推敲"。

**用户成功指标体系：**

| 指标类别 | 指标名称 | 目标值 | 衡量方式 |
|---------|---------|--------|---------|
| **质量** | 结果采纳率 | >80% | 用户直接采纳或少量修改的任务占比 |
| **效率** | 任务完成率 | >70% | 看板 Done/Total |
| **自主性** | 人工干预率 | <30% | 需要用户介入的任务占比 |
| **进化** | 规则库增长 | >20 条/月 | rules.md 新增规则数 |
| **价值** | 时间节省 | >5 小时/周 | 用户自我估算 |

**用户行为指标：**

| 行为 | 含义 | 正向信号 |
|------|------|---------|
| 持续使用 | 产品有价值 | 每周至少部署 3 个任务 |
| 任务升级 | 信任度提升 | 任务复杂度逐渐提高 |
| 主动扩展 | 深度采纳 | 主动添加新类型任务 |
| 规则审查 | 参与进化 | 定期查看/优化规则库 |

**情感成功标准：信任**

用户对 AI 分身产生信任感，愿意将重要任务交给它处理，不再需要反复检查每一个输出。

## MVP Scope

### Core Features

**4天交付，验证核心价值**

| 模块 | 功能 | 说明 |
|------|------|------|
| **Agent CLI 集成** | 外部 Agent 调度 | 封装 OpenCode、Claude Code、Gemini CLI 等工具 |
| **任务调度器** | 智能任务分配 | 从 kanban 读取任务，根据类型选择最适合的 Agent |
| **Soul 注入系统** | 个性化基础 | 读取 profile.md/rules.md/mission.md，拼接为 system prompt |
| **养蛊循环** | 自进化机制 | 执行结果 → 反思分析 → 规则提取 → 自动更新 rules.md |
| **Agentic RAG** | 经验复用 | 执行历史向量化存储，相似任务经验检索 |
| **Web Dashboard 升级** | 可视化管理 | 任务创建表单、Agent 状态展示、Soul 在线编辑 |

**核心工作流：**
```
用户提交任务 → Soul评估 → RAG检索相似经验 → Agent CLI执行 → 养蛊反思 → 规则库更新
```

**第一个验证任务：** 使用现有项目进行市场调研或技术分析

### Out of Scope for MVP

**Phase 2（验证成功后）：**

| 功能 | 说明 | 推迟原因 |
|------|------|---------|
| 多用户支持 | 团队协作功能 | MVP 专注单用户验证 |
| 权限管理 | 复杂权限系统 | 增加复杂度，非核心价值 |
| 移动端适配 | 手机端界面 | Web 端验证后再扩展 |
| MCP 集成 | Model Context Protocol | 技术复杂度高，v2.1 考虑 |
| 高级审批策略 | 复杂审批工作流 | 简单审批机制足够 |

**明确边界：**
- 不实现自定义 AI coding 能力（调用现有工具）
- 不支持多机器人协作（单 RDK X5 验证）
- 不包含复杂的 UI/UX 设计（功能优先）

**验证成功标准：**

| 标准 | 指标 | 达标条件 |

**Go/No-Go 决策点：**
