# Product Brief: AI-as-Me

## Executive Summary

AI-as-Me 是一个开源的自进化 AI 数字分身系统，旨在为 AI 创业者打造一个真正能"替您打工"的智能伙伴。核心价值主张：**让您的 AI 分身每周完成 X 个任务，节省 Y 小时**。

系统通过"养蛊"机制实现持续进化：看板任务 → 混合式澄清 → 执行（尝试+试错） → 记录日志 → 反思总结 → 更新规则库 → 下次更聪明。

优先部署平台为 RDK X5 机器人（XLeRobot），Day 1 即包含完整的机器人软件包部署，让数字分身既能处理商业分析，也能在物理世界执行任务。

商业模式：开源核心框架 + 卖结果（预配置硬件套件、企业版支持）。

## Core Vision

### Problem Statement

AI 创业者面临一个核心矛盾：他们需要具备多种角色能力（商业分析、技术开发、市场运营等），但个人时间和精力有限。现有的 AI 工具虽然智能，却无法真正"替人打工"——它们没有记忆、不了解用户的偏好和做事风格、只能被动回应而非主动行动。

### Problem Impact

- **效率损失**：每次与 AI 对话都要重新解释背景和偏好
- **能力瓶颈**：AI 无法积累经验，永远停留在"新手"状态
- **机会成本**：创业者被日常琐事消耗，无法专注于高价值决策
- **规模受限**：个人能力无法复制，业务增长受制于个人时间

### Why Existing Solutions Fall Short

| 现有方案 | 核心缺陷 |
|---------|---------|
| ChatGPT/Claude | 无持久记忆，不了解用户个性 |
| AutoGPT/AgentGPT | 不稳定，无个性化，无进化机制 |
| 个人知识库 + RAG | 只能检索，不能行动 |
| n8n/Zapier | 无智能，规则固定，无法适应变化 |
| Open Interpreter | 无长期记忆，无进化机制 |

**核心缺失：**
1. **无灵魂** - 不具备用户的履历、价值观、决策风格
2. **无进化** - 不能从经验中学习，越用越强
3. **无能动性** - 只能被动回应，不能主动推进目标

### Proposed Solution

AI-as-Me 通过三大核心机制解决这些问题：

**1. 灵魂注入 (Soul Injection)**
- `profile.md`：您的履历、技能栈、性格特征
- `rules.md`：从经验中提炼的决策法则
- `mission.md`：长期使命和目标

**2. 养蛊进化循环 (Self-Evolution Loop)**
```
看板任务 → 混合式澄清 → AI执行（尝试+试错） → 记录日志 → 反思总结 → 更新规则库 → 下次更聪明
```

**3. 混合式澄清机制 (Hybrid Clarification)**
- 简单任务：直接执行，遇到问题再询问
- 复杂任务：先与用户澄清需求，确认后再执行
- 高风险操作：必须人工审批

**优先部署平台：RDK X5 具身版 (XLeRobot)**

Day 1 部署清单：
- Phase 0: 基础环境（系统镜像、SSH、Python 环境）
- Phase 1: XLeRobot 软件包（核心库、驱动验证）
- Phase 2: AI Agent 系统（看板、agent.py、灵魂文件）
- Phase 3: 集成验证（端到端流程确认）

**集成能力：**
- 看板系统 (Kanban) - 文件级任务管理
- n8n 工作流自动化 - Phase 2 集成

### Key Differentiators

1. **"养蛊"进化机制** - 独创的自我学习循环，让 AI 越用越强
2. **混合式澄清** - 平衡自主性与可控性的智能交互模式
3. **具身优先** - 直接部署在 RDK X5，云端智能 + 物理执行
4. **开源核心** - MIT/Apache 2.0 协议，社区友好
5. **卖结果不卖工具** - 聚焦"节省时间、完成任务"的价值主张

### Business Model

**开源 + 商业化路径：**
1. **开源核心框架** - GitHub 社区建设、品牌积累
2. **商业化层**：
   - 预配置 RDK X5 套件（硬件 + 预装软件）
   - 企业版（私有部署 + 技术支持）
   - 预训练"灵魂"模板（行业特定规则库）

### Validation Plan

**第一个验证任务：bettafish 项目市场调研**

**30 天验证目标：**
| 指标 | 目标 |
|------|------|
| 任务完成率 | >70% |
| 规则库增长 | >20 条新规则 |
| 人工干预率 | <30% |
| 时间节省 | >5 小时/周 |

## Target Users

### Primary Users

**用户画像：技术型单人 AI 创业者**

**代表人物：Jody**
- **背景**：Python 开发者，专注 AI 领域
- **创业状态**：单人创业，身兼多职
- **技术栈**：Python、AI/ML 开发、机器人/嵌入式（RDK X5）

**日常工作：**
- 📊 AI 市场调研 - 发现创业机会
- 💻 写代码 - 开发 AI 产品和原型
- 📈 做运营 - 产品推广和用户增长
- 🔍 商业分析 - 评估市场和竞争态势

**核心痛点：**
- 单人创业需要扮演多个角色（开发者、产品经理、市场、商业分析）
- 时间被分散在各种任务中，难以专注于高价值工作
- 每次使用 AI 工具都要重新解释背景和偏好
- 缺乏一个能持续学习、了解自己的智能助手

**首要需求：**
- 将**商业分析**任务交给 AI 分身
- 利用项目（如 SmartFish-V1）进行市场调研，发现创业机会
- 分身能产出**经得起推敲**的高质量结果

**成功标准：**
- 任务完成结果质量高，经得起推敲
- 无需反复解释背景，分身"懂我"
- 规则库不断积累，分身越用越强

### Secondary Users

**潜在次要用户群体：**

| 用户类型 | 特征 | 需求差异 |
|---------|------|---------|
| **小型 AI 团队技术负责人** | 2-5 人团队，需要提升效率 | 可能需要多用户协作功能 |
| **AI 爱好者/学习者** | 技术背景，想尝试 AI Agent | 更注重学习和探索 |
| **开源社区贡献者** | 开发者，对"养蛊"机制感兴趣 | 贡献代码、扩展功能 |

**决策者：** 用户本人（单人创业/技术负责人直接决策）

### User Journey

**1. 发现阶段 (Discovery)**
- 通过技术博客/播客听说"AI 养蛊"或"AI 数字分身"概念
- 被"越用越强"的进化机制吸引
- 在 GitHub 发现 AI-as-Me 开源项目

**2. 首次使用 (Onboarding)**
- 按照 Day 1 部署清单在 RDK X5 上部署系统
- 上传"灵魂文件"（profile.md、rules.md、mission.md）
- 在看板中创建第一个任务：使用 SmartFish-V1 项目进行市场调研

**3. 核心使用 (Core Usage)**
```
每日循环：
1. 早上：在看板中部署新任务
2. AI 分身：混合式澄清 → 执行任务 → 记录日志
3. 晚上：查看完成结果，AI 自动反思更新规则库
4. 周末：审查规则库增长，见证分身进化
```

**4. "Aha！"时刻 (Value Realization)**
- 第一次看到分身完成的商业分析报告，**结果经得起推敲**
- 发现规则库自动积累了有价值的经验法则
- 意识到分身真的"懂我"了，不再需要反复解释

**5. 长期价值 (Long-term)**
- 分身成为日常工作的一部分
- 规则库成为个人知识资产
- 从"省时间"升级为"扩展能力边界"
- 考虑将成功经验分享给社区或商业化

## Success Metrics

### User Success Metrics

**核心成功标准：结果采纳率 > 80%**

AI 分身产出的结果（商业分析报告、市场调研等）被用户直接采纳或仅需少量修改即可使用的比例超过 80%，即视为"经得起推敲"。

**用户成功指标体系：**

| 指标类别 | 指标名称 | 目标值 | 衡量方式 |
|---------|---------|--------|---------|
| **质量** | 结果采纳率 | >80% | 用户直接采纳或少量修改的任务占比 |
| **效率** | 任务完成率 | >70% | 看板 Done/Total |
| **自主性** | 人工干预率 | <30% | 需要用户介入的任务占比 |
| **进化** | 规则库增长 | >20 条/月 | rules.md 新增规则数 |
| **价值** | 时间节省 | >5 小时/周 | 用户自我估算 |

**用户行为指标：**

| 行为 | 含义 | 正向信号 |
|------|------|---------|
| 持续使用 | 产品有价值 | 每周至少部署 3 个任务 |
| 任务升级 | 信任度提升 | 任务复杂度逐渐提高 |
| 主动扩展 | 深度采纳 | 主动添加新类型任务 |
| 规则审查 | 参与进化 | 定期查看/优化规则库 |

### Business Objectives

**当前阶段：产品验证优先**

暂不设定商业目标，专注于验证核心价值主张：
- AI 分身能否产出高质量结果（采纳率 >80%）
- "养蛊"进化机制是否有效（规则库持续增长）
- 用户是否真正节省时间（>5 小时/周）

商业目标将在产品验证成功后（30-90 天）根据实际数据制定。

### Key Performance Indicators

**30 天验证期 KPIs：**

| KPI | 目标 | 优先级 | 衡量周期 |
|-----|------|--------|---------|
| 结果采纳率 | >80% | P0 | 每任务 |
| 任务完成率 | >70% | P0 | 每周 |
| 规则库增长 | >20 条 | P1 | 30 天累计 |
| 人工干预率 | <30% | P1 | 每周 |
| 时间节省 | >5 小时/周 | P2 | 每周自评 |

**验证成功标准：**
- P0 指标全部达标 = 核心价值验证成功
- P0 + P1 指标达标 = 产品可进入扩展阶段
- 全部指标达标 = 可考虑开源发布和社区建设

## MVP Scope

### Core Features

**MVP 核心功能（Day 1 必须）：**

| 模块 | 功能 | 说明 |
|------|------|------|
| **文件级看板** | 任务管理 | inbox/todo/doing/done 目录结构 |
| **任务执行引擎** | agent.py | 核心执行循环，调用 LLM API |
| **混合式澄清** | 智能交互 | **MVP 阶段：全部先问用户确认** |
| **灵魂文件系统** | 个性化基础 | profile.md / rules.md / mission.md |
| **日志记录** | 执行追踪 | 任务执行过程的结构化记录 |
| **反思模块 V0.1** | 进化机制 | **手动触发反思**（V0.2 再做自动反思）|
| **规则更新** | 知识积累 | **建议模式**（AI 建议，用户确认后写入）|
| **XLeRobot 基础包** | 硬件支持 | 依赖文档：github.com/Vector-Wangel/XLeRobot |

**MVP 核心循环：**
```
看板任务 → 全部先澄清 → AI执行 → 记录日志 → 手动触发反思 → 建议规则 → 用户确认写入
```

**第一个验证任务：** 使用 SmartFish-V1 项目进行市场调研

**关键依赖：**
- XLeRobot: https://github.com/Vector-Wangel/XLeRobot
